# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- Короткая тематическая база знаний (10 документов) — учебные заметки и объяснения по retrieval/эмбеддингам.
- Чанкинг: простая разбивка по словам с фиксированным размером чанка и overlap.
- Векторизация: TF-IDF (фоллбек); при наличии Sentence-Transformers можно заменить на dense-эмбеддинги.
- Проверка retrieval: рассчитаны `hit@3` и keyword-based recall; среднего `hit@3` в бенчмарке достигнуто 1.0.
- Обновление базы знаний: добавлены 2 документа, показан эффект переиндексации (до/после).
- Mini-RAG: собран контекст из top-k чанков и сформирован ответ с указанием источников.

## 2. Среда и воспроизводимость

- Python: стандартная учебная среда (используются `numpy`, `pandas`, `scikit-learn`).
- faiss / faiss-cpu: не использовался (TF-IDF-фоллбек реализован в памяти).
- sentence-transformers / transformers / sklearn: используется `scikit-learn` (`TfidfVectorizer`) — при желании можно подключить `sentence-transformers`.
- torch (если использовался): не использовался.
- Устройство (CPU/GPU): CPU.
- Seed: 42.
- Как запустить: открыть `homeworks/HW14/HW14.ipynb` и выполнить Run All.

## 3. База знаний и подготовка документов

- Предметная область: учебные заметки по retrieval, эмбеддингам и сопутствующим понятиям (FAQ-подобная коллекция).
- Источник документов: встроенная в ноутбук коллекция коротких текстов (10 документов).
- Число исходных документов: 10.
- Число чанков после разбиения: 17 (после добавления 2 документов — 19).
- Какой способ чанкинга использовался: простая разбивка по словам с шагом = chunk_size - overlap.
- Основные параметры чанкинга: `chunk_size=24`, `overlap=6`.
- Комментарий: небольшая тематическая база хорошо подходит для учебной демонстрации pipeline: чанки дают достаточный контекст для извлекающей генерации ответов и позволяют наглядно показать эффект переиндексации.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- Какая embedding-модель или способ векторизации использовались: `TfidfVectorizer` (TF-IDF) — детерминированный фоллбек.
- Какая метрика сходства использовалась: косинусное сходство (через нормализованные векторы и скалярное произведение).
- Какой индекс `FAISS` использовался: `FAISS` не применялся; используется хранение векторов в памяти и матричное умножение.
- Какой `top_k` использовался по умолчанию: `top_k = 3`.
- Что показали первые примеры retrieval: top-3 для контрольных запросов в большинстве случаев возвращали ожидаемые тематические фрагменты.

### 4.2. Контрольные запросы и оценка retrieval

- Сколько контрольных запросов использовалось: 5 (малый учебный набор в `qa_benchmark`).
- Что считалось релевантным ответом: попадание хотя бы одного чанка из релевантного документа в топ-k.
- Какие метрики retrieval считались: `hit@k` (k=3) и keyword-based recall для текстовых ответов.
- Итоговый `hit@3`: среднее по бенчмарку = 1.0.
- Итоговый `recall@3`: приведён в `artifacts/retrieval_eval.csv` вместе с keyword_recall для baseline и RAG.
- Дополнительные метрики: keyword-based recall для baseline vs RAG (см. CSV).
- Какие запросы оказались самыми простыми: запросы с явными ключевыми словами (например про overlap и переиндексацию).
- Какие запросы оказались самыми проблемными: более абстрактные формулировки, где TF-IDF даёт менее однозначную выдачу.

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- Какой параметр сравнивался: в демонстрации использованы `chunk_size` и `overlap` как основные параметры (рабочий набор: `chunk_size=24`, `overlap=6`).
- Какие два варианта сравнивались: в учебной части фиксирован один рабочий набор; предлагается сравнить меньший/больший `chunk_size` или разные `overlap` для углублённого анализа.
- Как изменилось качество retrieval: выбранный набор параметров даёт хороший `hit@3`; увеличение overlap повышает вероятность попадания релевантной информации на границах чанков.
- Какой вариант вы выбрали как основной и почему: `chunk_size=24`, `overlap=6` как компромисс между информационной насыщенностью чанка и сохранением контекста.
- Что именно было обновлено в базе знаний: добавлены 2 документа (`doc_11`, `doc_12`) с целевыми фразами для демонстрации изменения выдачи.
- Как была выполнена переиндексация: повторный чанкинг и построение TF-IDF-векторов (в памяти) для новой коллекции.
- Что изменилось в retrieval после обновления базы знаний: для некоторых запросов изменился состав top-3 — подробности в `artifacts/retrieval_before_after_update.csv`.

## 7. Анализ

Выбранная база знаний оказалась удачной для учебной демонстрации: небольшая, тематически связная коллекция позволила проверить этапы pipeline и увидеть влияние retrieval на итоговые ответы. Чанкинг с `chunk_size=24` и `overlap=6` дал рабочий баланс — чанки информативны, а overlap помогает не терять смысл на границах. Формальная оценка (`hit@3`, keyword-based recall) дала количественное подтверждение корректности настроек: средний `hit@3` по набору = 1.0. Добавление двух документов продемонстрировало эффект переиндексации — некоторые запросы стали возвращать другие верхние кандидаты. Основные ошибки mini-RAG связаны с недостаточным retrieval (релевантный фрагмент не попал в top-k) и с простотой извлекающего генератора; в реальных задачах целесообразно сочетать dense-эмбеддинги и reranking.

## 8. Итоговый вывод

Для учебных задач оптимальным базовым конвейером является гибрид: TF-IDF для быстрой фильтрации и dense-эмбеддинги (Sentence-Transformers) + FAISS для точного ранжирования при увеличении масштаба. Главное наблюдение: качество retrieval ограничивает качество mini-RAG, поэтому инвестиции в представления и индекс часто дают больший эффект, чем усложнение шага генерации.
